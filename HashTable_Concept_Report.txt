HashTable Concept – Report

(1) Application
In this project, the generic templated `HashTable<K, V>` is used inside `ResourceManager` to manage game assets such as textures and animations. In `ResourceManager.h`, two instances are declared:
- `HashTable<std::string, std::unique_ptr<sf::Texture>> textureMap;`
- `HashTable<std::string, std::unique_ptr<Animation>> animationMap;`

These hash tables allow the game to load each resource once at startup and then retrieve it in constant average time using a string key (for example, `"floor"`, `"player_run"`, `"meteor_3"`). This is critical for a real‑time SFML game where performance matters and the same assets are requested many times per second.

The main operations are:
- During initialization, `ResourceManager::ResourceManager()` constructs `textureMap(32)` and `animationMap(16)` with initial bucket counts.
- `loadTextures()` and `loadPlayerAnimations()` create `std::unique_ptr<sf::Texture>` or `std::unique_ptr<Animation>` objects, call `loadFromFile(...)` or `animation.load(...)`, and then insert them into the hash tables via `textureMap.insert(name, std::move(texture));` and `animationMap.insert(name, std::move(anim));`.
- When gameplay code needs a resource, it calls `getTexture(name)` or `getAnimation(name)`. These methods call `textureMap.get(name)` or `animationMap.get(name)` and return the dereferenced `std::unique_ptr`, so game entities can draw sprites or play animations without reloading files.
- If a key is missing (for example, a typo in the name), `getTexture` and `getAnimation` catch the `std::runtime_error` thrown by `HashTable::get` and print an error message such as `ERROR: Texture 'xyz' not found in hash table!`, then fall back to a safe default (`"floor"` or `"player_idle"`).

In summary, the hash table is used as a fast lookup table for resources, supporting quick insertions during loading and frequent lookups during gameplay, while also providing a simple error‑handling path for missing keys.


(2) Concept
The `HashTable` class implements a generic associative container that maps keys of type `K` (here, `std::string`) to values of type `V` (here, `std::unique_ptr<sf::Texture>` and `std::unique_ptr<Animation>`). It uses the following ideas:
- It computes a bucket index with `std::hash<K>` and `index = hasher(key) % capacity`.
- Each bucket is a singly linked list of `Node` objects (`key`, `value`, and `Node* next`), implementing separate chaining to resolve collisions.
- The table automatically resizes by doubling `capacity` when the load factor `count / capacity` exceeds `loadFactorLimit`, and rehashes all elements into the new bucket array.

This design fits the resource‑manager use case because:
- Average O(1) insert and lookup time keeps resource access fast even as the number of textures and animations grows.
- Keys are human‑readable strings, which makes the calling code more understandable (`getTexture("fireball")` is clearer than using integer IDs).
- Using `std::unique_ptr` as the value type ensures single ownership of each SFML texture or animation, preventing accidental copies and ensuring that resources are freed when the hash table is cleared or destroyed.

Possible alternatives and reasons they were not used:
- `std::map<std::string, V>` or `std::multimap` could store the resources, but they are tree‑based and provide O(log n) lookups, which is slower than the expected O(1) of a hash table. They also do not demonstrate a custom hash table implementation, which is part of the data‑structure learning goal.
- `std::unordered_map<std::string, V>` has a similar hash‑table interface and could replace this class in production code. However, the assignment’s purpose is to implement the behavior manually, control memory management, and understand how hashing and collision handling work internally.
- A simple `std::vector<std::pair<std::string, V>>` could store resources, but lookups would be O(n) and would not scale well when many textures and animations are loaded.

(4) Troubleshooting Summary
Issues encountered and solutions
- Collision handling and resizing: Because the table uses chaining with linked lists, it was important to re‑insert all existing elements when resizing. The `resize()` function temporarily stores the old table, doubles the capacity, and reinserts all nodes using `insert(current->key, std::move(current->value));` to ensure that every element lands in the correct new bucket. This solved issues where keys would otherwise end up in the wrong bucket after a capacity change.
- Memory management: Each bucket stores `Node` objects allocated with `new`, and the values are `std::unique_ptr` objects owning SFML textures/animations. To avoid memory leaks, `clear()` walks each bucket’s linked list, deletes each `Node`, and then sets the bucket pointer to `nullptr`. The destructor and copy‑assignment operator both use `clear()` and rebuild the table as needed, which fixed early problems with double‑free and missing‑free bugs.
- Exception safety and error reporting: Directly using `get` to access missing keys would throw an exception and potentially crash the program. Wrapping `textureMap.get(name)` and `animationMap.get(name)` inside a `try/catch` block in `getTexture` and `getAnimation`, and printing a clear error message with a fallback resource, made the system more robust and easier to debug.
- Interface design: Providing both `insert` and `operator[]` made the class easier to use in different situations, but for resource loading, `insert` with a move‑only `std::unique_ptr` is preferred. This design avoids accidental creation of empty default resources and prevents copying SFML objects.
