1a. Inheritance and Derived Classes (2–3 minutes)

So these things in the game - the player, the enemy, the platform, the meteor, the hp potion they are all entities in the game. we can see this in the UML class diagram.

The game's architecture is built on a three-level inheritance hierarchy. At the base level, the Entity class defines shared features: position, velocity, rendering, an activation flag, and a pure virtual update method that must be implemented by all these derived classes. Entity also declares getBounds as virtual, allowing derived classes to override collision detection.

Character extends Entity and adds attributes for combat—health, damage values, movement speed, and jumping mechanics—along with virtual methods such as takeDamage and isAlive. Character is still abstract because it doesn't implement the pure virtual update method.

Player builds on Character with input handling, physics, animations, special abilities, a leveling system, and a skill tree. Player implements the pure virtual update method and overrides getBounds to provide player-specific collision detection. Enemy extends Character to implement AI—patrolling, chasing, and attacking—while Boss introduces multi-phase behavior.

Projectiles follow a similar structure. A base Projectile class extends Entity and defines shared behavior, while FireProjectile, IceProjectile, and Meteor override update and getBounds to deliver their elemental effects and custom collision boxes.

This three-level hierarchy—Entity → Character → Player—demonstrates how inheritance allows all game objects to share core capabilities while providing room for specialized behaviors at each level.

1b. Polymorphism (2–3 minutes)

[SHOW: Code snippet from CombatSystem.cpp showing baseProjectile->update(dt) and baseProjectile->getBounds()]
[SHOW: Code snippet from GameWorld.cpp showing platformEntity->update(dt)]

Polymorphism allows different game entities to be managed uniformly. By calling virtual functions through base-class pointers, the game loop can treat Players, Enemies, and Projectiles in the same way while still invoking their correct behaviors at runtime. 

For example, in the combat system, I store FireProjectile and IceProjectile objects but call update and getBounds through a Projectile base pointer. This eliminates the need for manual type checking—without polymorphism, I would need if-else chains checking entity types and casting to specific classes. Instead, when update is called, each entity executes its own logic—FireProjectile animates fire effects, IceProjectile applies slow effects, Meteors fall with gravity. The same applies to getBounds—each projectile type returns its own custom collision box.

This design supports a large variety of behaviors while preserving a unified interface.

Data Structures Overview (1–2 minutes)

[SHOW: Diagram or overview slide of all six data structures]

The game uses six custom data structures, each chosen for a specific real-time requirement.

A custom hash table powers the ResourceManager class, giving constant-time access to the textures and sounds when the game loads sprites or plays audio. A singly linked list stores the combat log in chronological order, supporting fast insertion as events occur. A doubly linked list manages object pools such as projectiles, potions, and meteors, enabling efficient addition and removal during gameplay. A stack manages game states (menus, pause, skill tree), ensuring states are pushed and popped in a strict LIFO order. A queue controls floating damage text, guaranteeing that damage numbers appear in the sequence they were created. Finally, an N-ary tree—a tree where each node can have any number of children—represents the skill system, naturally enforcing prerequisite relationships between abilities.

These structures ensure that resource lookup, state transitions, combat event handling, visual feedback, and skill-tree traversal all remain efficient during gameplay. Every core operation involved—insert, lookup, push, pop, enqueue, dequeue—executes in constant time, which is essential for maintaining a smooth gameplay experience.

2a. Hash Table – Independent Research (3–4 minutes)

[SHOW: Code snippet of Node structure - lines 19-30 showing key, value, next pointer]
[SHOW: Code snippet of hash() function - lines 107-111 using std::hash]
[SHOW: Code snippet of insert() method - lines 230-258 showing collision handling and resize trigger]
[SHOW: Code snippet of resize() method - lines 114-146 showing capacity doubling and rehashing]

The hash table is a fully custom implementation using separate chaining for collision resolution. Each bucket contains a linked list of Node structures, where each Node stores a key, value, and a next pointer to handle collisions.

The hash function uses std::hash to compute a hash value, then mods by capacity to get the bucket index. When inserting, the method first checks if the key already exists in the bucket's chain—if so, it updates the value. If not, it creates a new node and inserts it at the head of the chain.

After each insertion, the table checks if the load factor exceeds 0.75. If it does, the resize method doubles the capacity, allocates a new bucket array, and rehashes all existing entries into the new table. This maintains low collision rates and ensures O(1) average-case performance for lookup, insert, and remove operations.

The hash table is used heavily in the ResourceManager to store textures and animations, allowing the game to retrieve assets instantly during rendering and gameplay.

2b. Singly Linked List (2 minutes)

[SHOW: Code snippet of Node structure - lines 14-21 showing data and next pointer]
[SHOW: Code snippet of pushBack() method - lines 133-147 showing O(1) tail insertion]
[SHOW: Code snippet of copy-and-swap idiom - lines 88-96 showing exception-safe assignment]

[SHOW: Combat log displaying in-game, updating in real-time]

The singly linked list stores combat log entries in chronological order. Each Node contains data and a single next pointer, forming a forward-only chain. The list maintains both head and tail pointers, enabling O(1) insertion at both ends.

The pushBack method demonstrates O(1) tail insertion—if the list is empty, both head and tail point to the new node; otherwise, the current tail's next pointer is updated, and tail moves to the new node. This makes appending combat events extremely efficient.

For memory safety, the list implements the Rule of Five: copy constructor, copy assignment using copy-and-swap idiom, move constructor, move assignment, and destructor. The copy-and-swap technique creates a temporary copy first, then swaps pointers—if the copy fails, the original list remains unchanged, providing strong exception safety.

This structure is ideal for the combat log because entries arrive in chronological order and typically only need to be appended or trimmed from the front.

The doubly linked list provides bidirectional traversal with each node storing value, next, and previous pointers. It uses a NIL sentinel node—a static instance representing the end of the list.

The pushBack method updates both forward and backward links. The new node's next points to NIL (the sentinel), and its previous points to the current tail. The previous tail's next pointer is then updated to point to the new node, completing the bidirectional linkage.
The erase method demonstrates O(1) removal when given an iterator. It updates the previous node's next pointer to skip over the current node, and the next node's previous pointer to point back to the previous node, effectively unlinking the node from the list. For object pooling, instead of deleting the node, we return it to the pool—this allows us to remove inactive projectiles during iteration and reclaim them for reuse without memory reallocation.

When spawn is called, pop_front removes a projectile from the pool, then it's added to the fireProjectiles doubly linked list, transitioning it to the active state. Memory stays allocated throughout the object's lifetime—we're just moving nodes between lists.

Active projectiles move to the processing state each frame where update(dt) runs their logic. If isActive returns true, they continue processing and rendering. When a projectile hits a target or goes out of bounds, it transitions to inactive. The reclaim method erases it from the active list in O(1) time using the iterator, then push_back returns the node to the pool for reuse.

The doubly linked list is essential here because during the forEach iteration over active projectiles, we can safely erase nodes and return them to the pool without breaking iteration. 3a. Stack (2–3 minutes)

[SHOW: Code snippet of Node structure - lines 36-46 showing data and next pointer]
[SHOW: Code snippet of push() method - lines 111-117 showing O(1) insertion at top]
[SHOW: Code snippet of pop() method - lines 131-141 showing O(1) removal from top]

[DEMO: Navigate through menus - main menu → pause → skill tree, then back out]

The stack manages game states such as menus, pause screens, and the skill tree using a singly linked list. Each Node contains data and a next pointer to the node below it. The topNode pointer always references the most recently added element.

The push method demonstrates O(1) insertion—it creates a new node, links it to the current top, and updates topNode. This makes state transitions instant. The pop method removes the top element in O(1) time by moving topNode to the next node and deleting the old top.

A unique feature is the peek method, which allows inspecting elements deeper in the stack without removing them. It traverses downward from the top by following next pointers. This is useful for checking if a pause menu is already open before pushing another state.

A specialized debug logger tracks push and pop operations for GameStateData, outputting state names during development. The stack ensures states exit in the reverse order they were entered, matching the natural structure of nested menus.

3b. Queue (2 minutes)

[SHOW: Code snippet of Node structure - lines 36-46 showing data and next pointer]
[SHOW: Code snippet of enqueue() method - lines 138-156 showing O(1) rear insertion]
[SHOW: Code snippet of dequeue() method - lines 160-175 showing O(1) front removal]
[SHOW: Code snippet of specialized debug logger - lines 87-101 showing FloatingText logging]

[DEMO: Attack enemies to show floating damage text appearing in sequence]

The queue controls floating damage text using a singly linked list with both front and rear pointers. This enables O(1) operations at both ends—critical for real-time gameplay.

The enqueue method adds elements to the rear in constant time. If the queue is empty, both frontNode and rearNode point to the new node; otherwise, the current rear's next pointer is updated, and rearNode moves to the new node. This ensures damage numbers are added instantly without frame drops.

The dequeue method removes elements from the front in O(1) time, maintaining FIFO ordering. When the last element is removed, both frontNode and rearNode are set to nullptr, resetting the queue to empty.

A specialized debug logger for unique_ptr<FloatingText> outputs the actual damage value being enqueued or dequeued, making it easy to verify that damage numbers appear in the correct sequence during testing.

3c. N-ary Tree (2–3 minutes)

[SHOW: Code snippet of NTree structure - lines 12-16 showing key, nodes array, parent pointer, and NIL sentinel]
[SHOW: Code snippet of attachNTree() method - lines 149-161 showing parent-child linking]
[SHOW: Code snippet of search() method - lines 201-217 showing depth-first search]
[SHOW: Code snippet of traverseLevelOrder() - lines 223-297 showing BFS with internal queue]

[SHOW: Skill tree UI with parent-child relationships highlighted]

The N-ary tree powers the skill system. Each node contains a key (the skill data), a fixed-size array of N child pointers, and a parent pointer. The tree uses a NIL sentinel—a static instance representing empty nodes—which eliminates null checks and simplifies edge cases.

The attachNTree method links a child to a specific index. It verifies the slot is empty (points to NIL), then installs the child and updates its parent pointer. The detachNTree method reverses this, removing the child and clearing its parent pointer, allowing subtrees to be reorganized dynamically.

The search method uses depth-first traversal to locate skills. It checks the current node's key, then recursively searches each child subtree, returning a pointer to the matching node or nullptr if not found.

The traverseLevelOrder method implements breadth-first search using an internal queue. It visits nodes level-by-level, which is perfect for rendering the skill tree UI in layers. The internal queue is a minimal implementation that stores tree node pointers and supports push, pop, and empty operations.

This structure naturally represents hierarchical dependencies—parent skills must be unlocked before children become available.

4. Design Patterns
4a. Iterator Pattern (2 minutes)

[SHOW: Code snippet of DoublyLinkedNodeIterator class - lines 12-46 showing iterator structure]
[SHOW: Code snippet of operator++() - lines 84-89 showing prefix increment]
[SHOW: Code snippet of operator--() - lines 104-108 showing prefix decrement]
[SHOW: Code snippet of operator*() and operator->() - lines 66-80 showing dereference operations]
[SHOW: Code snippet of operator== and operator!= - lines 121-131 showing comparison]

The DoublyLinkedList is used for object pooling throughout the game because of its bidirectional traversal capability - the next and previous pointers enable fast O(1) removal from any position in the list. Additionally, the Iterator design pattern built into this structure provides both external and internal iteration mechanisms, making it highly versatile for game systems that need to frequently add, remove, and traverse active objects like projectiles and hp potions.

So, speaking about the Iterator design pattern in my program, it appears in two forms.

External iterators, such as the DoublyLinkedNodeIterator, provide STL-compatible traversal. The iterator class wraps a node pointer and overloads standard operators. The prefix increment operator (++) moves to the next node by calling getNext(), while the prefix decrement operator (--) moves backward via getPrevious(). This bidirectional capability is essential for the doubly linked list.

The dereference operator (*) returns a reference to the node's stored value, and the arrow operator (->) returns a pointer, enabling direct member access. Equality comparison checks if two iterators point to the same node, which is critical for loop termination when comparing against end().

The combination of begin and end function in the DoublyLinkedList container together with these operators ++, * and != in the iterator is what allows range-based for loops to work seamlessly with custom data structures. This can be seen in Session.cpp. 

Internal iterators, via forEach methods, let structures apply a function to each element without exposing traversal logic. The client passes a function to apply to each element and the data structure handles iteration internally. This is used extensively for updating and rendering entities across my game.

Now let's look at the UML diagram for Iterator Design Pattern. The UML diagram illustrates the relationships between the three core components. DoublyLinkedList contains DoublyLinkedNode objects, which form the underlying data structure. The list creates DoublyLinkedNodeIterator instances through its begin() and end() methods. The iterator maintains a pointer to a node, enabling it to traverse the linked structure.

Crucially, the iterator is declared as a friend of DoublyLinkedList, granting it access to internal node pointers while keeping them hidden from other client code. This friendship is essential because the iterator constructor takes a raw Node pointer, which only the list should provide.

4b. Singleton Pattern (2 minutes)

[SHOW: Code snippet of getInstance() method - Game.cpp lines 6-15 showing static local variable]
[SHOW: Code snippet of private constructor - Game.h lines 28-29]
[SHOW: Code snippet of deleted copy/move operations - Game.h lines 34-44]

The Singleton pattern ensures a single instance of the Game class. The getInstance() method uses a static local variable—when first called, it constructs the Game instance, and on subsequent calls, it returns the same instance. This is thread-safe in C++11 and later due to guaranteed initialization of static locals.

The constructor is private, preventing direct instantiation via `Game g;`. Copy and move operations are explicitly deleted using `= delete`, preventing accidental duplication through assignment or passing by value. This ensures there is exactly one Game object throughout the program's lifetime.

This pattern is essential for the Game class because it manages the window, resource manager, and active session. Multiple Game instances would create conflicting windows and duplicate resource loading. The Singleton provides a central access point—any part of the code can call `Game::getInstance()` to access the shared game state.

The ResourceManager is also accessed through this single Game instance, ensuring textures and sounds are loaded once and shared across all systems.
