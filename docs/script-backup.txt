Data Structures Overview (1-2 minutes)
Before we dive into the individual implementations, let me give you an overview of the six custom data structures I built and how they're used throughout the game.
I've implemented a hash table for resource management, which handles texture and sound lookup by string key with insert and lookup operations running in O(1) average time. Next, I have a singly linked list for combat log messages, which maintains sequential insertion and tracks procedural event logging with O(1) insert time and O(n) traversal. The doubly linked list manages projectile, potion, and meteor pools, allowing bidirectional traversal with O(1) insertion at both ends. For game state management, I use a stack following the LIFO principle, which handles pushing and popping menus with O(1) operations. The queue, following FIFO ordering, manages floating damage text to ensure temporal ordering of procedural events, also with O(1) enqueue and dequeue operations. Finally, the N-ary tree represents the skill tree with its hierarchical dependencies, offering O(log n) performance for balanced operations.
Let me explain why I chose these specific structures for each use case. I selected a hash table for the ResourceManager because the game needs instant access to textures and sounds by name. When the game needs to load a player sprite or play an attack sound, it can't wait - it requires O(1) average lookup time. The hash table maps string keys like 'player_idle' directly to the texture data using separate chaining for collision resolution.
For the combat log, I need to maintain chronological order of events from oldest to newest. A singly linked list is perfect for this purpose because I only insert at the tail for new events and occasionally traverse from head to tail to display the log. Since I don't need to remove from the middle or traverse backwards, the simpler singly linked structure is more efficient than its doubly linked counterpart.
In contrast, the projectile pool requires bidirectional traversal because I need to efficiently add and remove projectiles from both ends. When a projectile expires, I can remove it in O(1) time from anywhere in the list if I have an iterator to it. The doubly linked structure with previous pointers makes this possible.
Game state management naturally fits a stack structure. When you open the pause menu, it pushes a new state on top. When you resume, it pops that state off. When you open the skill tree from the pause menu, that's another push. The LIFO ordering ensures states are exited in reverse order of entry, which is exactly what we need for nested menus.
Floating damage numbers must appear in the order they were created - first damage dealt appears first, second damage dealt appears second. A FIFO queue guarantees this temporal ordering. When damage is dealt, it enqueues the text, and the rendering system dequeues and displays them in order.
The skill tree has hierarchical dependencies - you can't unlock 'Ice Projectile Level 2' without first unlocking 'Ice Projectile Level 1'. An N-ary tree naturally represents this parent-child relationship. Each skill node can have multiple children representing skills that unlock after it, and I can traverse the tree to find unlockable skills based on what the player has already learned.
From a performance perspective, notice that all the core operations - insert, lookup, push, pop, enqueue, and dequeue - are O(1) constant time. This is critical for a real-time game running at 60 frames per second. If resource lookup was O(n), the game would lag every time it needed to load a texture. If state transitions were slow, menus would feel sluggish. Choosing the right data structure for each use case ensures the game runs smoothly.

1. Object-Oriented Programming
1a. Inheritance and Derived Classes (2-3 minutes)
First, let's talk about inheritance and how I use a class hierarchy to organize all game entities.
At the foundation, I have the Entity class which provides common functionality for all game objects. It contains position and velocity vectors for movement, a sprite for rendering, an active state flag to track whether the entity is currently in use, and a pure virtual update method that forces all derived classes to implement their own update logic. This pure virtual function is crucial because it establishes a contract that every entity must define how it updates itself each frame.
Building upon this foundation, the Character class inherits from Entity and adds combat-specific features. It introduces HP and maxHP for health management, damage values for combat calculations, movement speed for character motion, and ground detection for jumping mechanics. It also provides virtual methods like takeDamage and isAlive that can be overridden by derived classes to customize behavior.
The Player class extends Character further and adds numerous player-specific features. It includes an animation system with idle, run, and attack animations to provide visual feedback. Input handling processes keyboard controls for player movement and actions. The jump mechanics support double-jump capability for enhanced movement options. A special attack with dash movement adds combat depth. Most importantly, it incorporates an experience and leveling system that allows the player to grow stronger, along with a skill tree for ability customization and an inventory for HP potions.
Similarly, the Enemy class inherits from Character and adds AI-specific functionality. It implements AI behavior with patrol, chase, and attack states to create believable enemy movement patterns. Detection and attack ranges determine when enemies notice and engage the player. Status effects like ice stacks and burning create interesting combat interactions. The Boss class further extends Enemy with boss-specific mechanics including minion spawning, health regeneration, and multiple attack patterns to create challenging encounters.
I also have a projectile hierarchy that demonstrates inheritance in a different context. The Projectile base class defines common properties like damage and lifetime. FireProjectile inherits from this and applies burn damage over time to enemies. IceProjectile also inherits from the base class but instead slows and freezes enemies. Both inherit from Entity and override the update method to implement their specific behaviors.
This demonstrates the IS-A relationship perfectly: Player IS-A Character IS-A Entity. Through inheritance, I achieve significant code reuse, reducing duplication across the codebase. Each level of the hierarchy adds more specific functionality while maintaining the common interface established by its parent classes.
1b. Polymorphism (2-3 minutes)
Now let's examine polymorphism - the ability to treat derived classes through their base class interface, which is one of the most powerful features of object-oriented programming.
Notice how all my classes define virtual functions to enable polymorphic behavior. Entity has a pure virtual update method, making it an abstract class that cannot be instantiated directly. Character has virtual takeDamage and isAlive methods that derived classes can override. Each derived class provides its own implementation of these methods, creating specialized behavior while maintaining a common interface.
In my game loop, I can store different entity types in collections and call their methods polymorphically. When I call update on a Player pointer, it executes Player's update logic, handling input, animation, and player-specific behavior. When I call update on an Enemy pointer, it executes Enemy's AI logic, managing patrol routes, target detection, and attack patterns. The correct version is determined at runtime through virtual function dispatch, which is the magic of runtime polymorphism.
For projectiles, this pattern continues. FireProjectile's update method handles fire animation and applies burn effects to enemies on hit. IceProjectile's update method handles ice animation and applies freeze effects instead. Despite these differences, both can be stored as Projectile pointers and updated polymorphically. The game loop doesn't need to know the specific type of projectile - it just calls update, and each projectile executes its own specialized behavior.
This demonstrates runtime polymorphism through virtual functions, which allows treating different types uniformly through base class pointers. Each object knows its own type and executes the correct method automatically, eliminating the need for large switch statements or type checking throughout the codebase.

2. Composite Data Structures
2a. Hash Tables - Independent Research (3-4 minutes)
For my independent research component, I implemented a hash table from scratch to manage game resources efficiently. This was a substantial undertaking that required understanding both the theoretical foundations and practical implementation challenges of hash tables.
My HashTable is a templated class that uses separate chaining for collision resolution. Let me walk through the key components and how they work together.
The structure consists of an array of bucket pointers, where each bucket is a linked list of key-value pairs. I use std::hash to compute hash values from keys, and the table automatically resizes when the load factor exceeds 0.75 to maintain optimal performance. This dynamic resizing is crucial for maintaining O(1) average case performance as the number of elements grows.
I implemented several key features to make this hash table robust and efficient. The hash function uses std::hash and modulo operation to map keys to buckets, distributing entries across the available space. The insert operation computes the hash index, searches the chain for an existing key, and either updates the existing value or creates a new node at the bucket head. If this insertion causes the load factor to exceed the threshold, it triggers a resize operation automatically.
The resize operation is particularly interesting because it doubles the capacity and then rehashes all existing entries into the new table. This ensures that as the table grows, entries remain well-distributed across buckets. While resizing is an O(n) operation, it happens infrequently enough that the amortized cost remains O(1) for insertions.
The search and get operations traverse the appropriate chain to find a matching key. Because I use separate chaining, multiple keys that hash to the same index form a linked list, and I search through this list linearly. The remove operation handles deletion from the head, middle, or tail of a chain, properly maintaining the linked list structure in all cases.
I use hash tables extensively in my ResourceManager to store textures and animations, both mapped by name. This gives me O(1) average-case lookup time for loading sprites and animations during gameplay, which is essential for maintaining smooth performance. When the game needs to render the player's idle animation or load an enemy sprite, the hash table provides instant access without searching through arrays or lists.
I also implemented diagnostic methods to analyze hash table performance. The loadFactor method returns the current load factor, bucketCount reports the number of buckets, longestChainLength identifies the worst-case chain length for collision analysis, and averageChainLength calculates the mean chain length across all buckets. These diagnostics help verify that the hash function distributes keys effectively and that separate chaining handles collisions appropriately.
The implementation demonstrates separate chaining handling collisions with linked lists, dynamic resizing maintaining O(1) average performance, and real-world application through fast resource lookup by name. I also implemented proper copy and move semantics for correct memory management, ensuring the hash table behaves correctly when copied or moved.
2b. Singly Linked List (2 minutes)
I implemented a singly linked list to store combat log entries in chronological order, maintaining a record of game events as they occur.
My implementation includes a node structure with data and a next pointer, providing the basic building blocks of the list. I maintain both a head pointer pointing to the oldest element and a tail pointer pointing to the newest element, which enables O(1) insertion at both ends. A count variable tracks the size for efficient size queries.
The pushBack operation adds an element at the tail in O(1) time, maintaining chronological order. It creates a new node, then either links it to the current tail or sets it as the first element if the list was empty. The pushFront operation adds an element at the head, also in O(1) time, though I use this less frequently for the combat log.
The popFront operation removes the oldest element in O(1) time by saving the current head, advancing the head pointer to the next node, and then deleting the old node. This is perfect for removing old log entries when the log grows too long. The removeAt operation removes an element at a specific index in O(n) time, requiring traversal to find the target node.
I also implemented removeIf, which removes all elements matching a predicate. This is particularly useful for filtering old log entries based on conditions like timestamp or event type. The forEach method applies a function to each element, enabling clean iteration without exposing the internal node structure.
For proper memory management, I implemented the Rule of Five. The copy constructor creates a deep copy of the list, the copy assignment operator uses the copy-and-swap idiom for exception safety, and I provide move constructor and move assignment for efficient transfer of ownership. The destructor properly deallocates all nodes to prevent memory leaks.
This structure provides efficient O(1) insertion at both ends with the tail pointer, making it ideal for maintaining chronological order of events. The template-based design provides type flexibility, allowing me to create linked lists of any type throughout the codebase.
2c. Doubly Linked List (2 minutes)
I also implemented a doubly linked list, which allows bidirectional traversal and provides additional capabilities compared to its singly linked counterpart.
The key difference lies in the node structure. Each node has both a next pointer and a previous pointer, allowing traversal in both directions. I use a NIL sentinel node to represent empty positions and end markers, which simplifies edge case handling throughout the implementation.
This bidirectional capability provides several advantages. The popBack operation can remove from the tail in O(1) time, which is impossible with a singly linked list where you'd need O(n) traversal to reach the node before the tail. The operation saves the tail node, moves the tail pointer to the previous node, and updates the links appropriately.
Even more powerful is the erase operation, which can remove any element in O(1) time if you have an iterator to it. By updating the previous and next links, I can surgically remove a node from anywhere in the list without traversing from the beginning. This makes doubly linked lists ideal for situations where you need to remove elements during iteration.
To support this bidirectional traversal, I implemented a custom iterator that supports both forward iteration with operator++ and backward iteration with operator--. The iterator also provides dereference with operator* and arrow operator with operator->, making it compatible with range-based for loops and giving it an STL-like interface.
The forEach method demonstrates the Iterator pattern in action. It traverses the list and applies a function to each element, separating the iteration mechanism from the element processing logic.
Overall, the doubly linked list provides bidirectional traversal capability, O(1) removal from both ends, a custom iterator for an STL-like interface, and the NIL sentinel pattern that eliminates null checks throughout the code. These features make it ideal for managing collections where elements need to be added and removed from multiple positions efficiently.

3. Abstract Data Types
3a. Stack (2-3 minutes)
I implemented a stack ADT to manage game states, handling transitions between menu, playing, paused, and game over states. The stack follows the LIFO principle - Last-In-First-Out - which naturally models how game states nest and unwind.
The structure consists of a node with data and a next pointer, a topNode pointer indicating the current top of the stack, and a count variable for efficient size tracking.
The core operations demonstrate the LIFO principle clearly. The push operation adds an element to the top in O(1) time by creating a new node, linking it to the current top, updating the top pointer, and calling the debug logger. The pop operation removes the top element, also in O(1) time. It first checks if the stack is empty to avoid errors, then saves the old top, moves the top pointer down to the next element, calls the debug logger, and finally deletes the old node to free memory.
The top operation returns a reference to the top element without removing it, providing read access while maintaining the stack structure. It throws an exception if the stack is empty, ensuring safe access. The peek operation is particularly interesting because it allows access to elements deeper in the stack by level. It traverses down to the specified level and returns the element at that depth, which is useful for checking game state history without modifying the stack.
I created a specialized debug logger that prints when game states are pushed and popped. This template specialization for GameStateData helps track state transitions during development, making it easier to debug issues with menu navigation and game flow. This demonstrates the template specialization technique, where I provide custom behavior for specific types.
The LIFO ordering enforces proper state management automatically. When you open the pause menu, it pushes a new state. When you open the skill tree from the pause menu, that's another push. When you close the skill tree, it pops back to pause. When you resume the game, it pops back to playing. The stack guarantees states are exited in reverse order of entry, which is exactly what we need for nested menus and game state management.
All operations - push, pop, and top - run in O(1) time, making state transitions instant with no performance impact. The template-based design allows storing any type, though I primarily use it for game state data. I've also enabled move semantics while disabling copy operations for efficiency, since game states shouldn't be duplicated.
3b. Queue (2 minutes)
I implemented a queue ADT to manage floating damage text, ensuring numbers appear in FIFO order - First-In-First-Out. This maintains the temporal ordering of damage events, so players see damage numbers in the exact sequence they occurred.
The structure includes a node with data and next pointer, a frontNode pointer indicating the oldest element, a rearNode pointer indicating the most recently inserted element, and a count variable for size tracking.
The enqueue operation adds an element to the rear in O(1) time. It creates a new node, and if the queue was empty, it sets both front and rear pointers to this new node. Otherwise, it links the new node to the current rear and updates the rear pointer. Finally, it increments the count and logs the operation.
The dequeue operation removes an element from the front in O(1) time, maintaining FIFO order. It first checks if the queue is empty to avoid errors, then saves the front node, moves the front pointer forward to the next element, and updates the rear pointer if the queue becomes empty. After logging the operation, it deletes the node to free memory.
The front operation returns a reference to the front element without removing it. It throws an exception if the queue is empty and otherwise returns the frontNode's data. This allows peeking at the next damage number to be displayed without removing it from the queue.
The forEach method iterates through all elements from front to rear, applying a function to each. This is useful for rendering all active floating text or updating their positions. The removeIf operation removes elements matching a predicate, handling removal from the front, middle, or rear of the queue. This is helpful for removing damage text that has faded out or moved off screen.
I specialized the debug logger for unique pointers to FloatingText, so it prints the actual damage text when elements are enqueued or dequeued. This helps debug the damage display system during development.
The FIFO ordering ensures damage text appears in the correct order, which is critical for player feedback. Both enqueue and dequeue operations run in O(1) time, allowing the game to efficiently manage large numbers of floating text without performance degradation. This makes the queue perfect for managing temporary visual effects that need temporal ordering.
3c. Tree (2-3 minutes)
I implemented an N-ary tree ADT for the skill tree system, where each node can have multiple children representing skills that unlock after it.
My NTree is a generic N-ary tree with template parameters for the data type and the maximum number of children per node. The structure stores a key value at each node, maintains a fixed-size array of N child pointers, includes a parent pointer for upward traversal, and uses a NIL sentinel node representing empty children. This sentinel pattern simplifies edge case handling throughout the implementation.
The attachNTree operation attaches a subtree as a child at a specified index. It first validates the index and checks for an empty slot to avoid overwriting existing children. Then it updates the child pointer and sets the parent pointer in the attached subtree, maintaining the bidirectional parent-child relationship.
The detachNTree operation removes and returns a child subtree. It saves the detached pointer, replaces it with NIL in the parent's children array, and clears the parent pointer in the detached subtree. This allows restructuring the tree dynamically as players unlock or respec their skills.
The search operation recursively searches for a key using depth-first traversal. It returns nullptr if the current subtree is empty, checks if the current node matches the key, and then recursively searches all children if no match is found. This enables finding specific skills in the tree by their identifier.
The getSize operation recursively counts all nodes in a subtree. It returns 0 for NIL nodes and otherwise counts the current node plus all children recursively. This is useful for calculating how many skills a player has unlocked in a particular branch.
The traverseLevelOrder operation performs breadth-first traversal using an internal queue implementation. Rather than depending on an external queue class, I implemented a minimal internal queue to keep the tree self-contained. The traversal visits nodes level by level and applies a visitor function to each node's data. This is particularly useful for rendering the skill tree UI layer by layer, showing skills in the order they unlock.
The N-ary tree allows multiple children per node, which is essential for skill trees where one skill might unlock several others. The NIL sentinel pattern simplifies edge cases by eliminating null checks throughout the code. The tree supports both depth-first traversal through the search method and breadth-first traversal through level-order traversal, giving flexibility in how we navigate the skill relationships. Finally, parent pointers enable upward navigation, which is useful for checking if prerequisites are met when attempting to unlock a skill.

4. Design Patterns
4a. Iterator Pattern (2 minutes)
I implemented the Iterator pattern to provide a standard way to traverse my data structures, creating a uniform interface regardless of the underlying implementation.
My custom iterator for doubly linked lists provides several key operations. The dereference operator returns a reference to the current element's value. The arrow operator enables member access on the current element. The increment operator moves to the next element by following the next pointer. The decrement operator moves to the previous element by following the previous pointer. Finally, the equality and inequality operators compare iterators to determine if they point to the same position.
The list provides begin and end methods that return iterators. The begin method returns an iterator to the first element, while end returns an iterator to the NIL sentinel. This enables range-based for loops, allowing code like "for each item in myList" with clean, readable syntax that matches standard library containers.
I also implemented forEach methods in multiple data structures. SinglyLinkedList, DoublyLinkedList, Queue, and the NTree all have forEach methods that accept function objects and apply them to each element. This represents the internal iterator pattern, where the data structure controls iteration and the client provides only the operation to perform.
For example, you might write list.forEach with a lambda function that processes each element. The forEach method handles all the traversal logic internally, and your code focuses solely on what to do with each element.
This demonstrates two variations of the Iterator pattern. The external iterator, represented by DoublyLinkedNodeIterator, gives clients full control over iteration, allowing them to advance, retreat, or skip elements as needed. The internal iterator, represented by forEach methods, keeps control within the data structure and simply applies an operation to each element.
The pattern provides a uniform interface for traversing different data structures, enables range-based for loops and STL-like usage, and separates the iteration mechanism from element processing. Whether you're iterating through a list, queue, or tree, the same conceptual approach applies, making the codebase more consistent and easier to understand.
4b. Singleton Pattern (2 minutes)
Finally, I implemented the Singleton pattern to ensure only one Game instance exists throughout the program's lifetime.
The Singleton pattern ensures a class has only one instance and provides global access to it. This is critical for the Game class because having multiple game instances would cause conflicts with window management, resource loading, and game state.
My implementation includes a static getInstance method that returns the single instance. The constructor is private, preventing direct instantiation from outside the class. I've deleted the copy constructor and copy assignment operator to prevent copying, and I've also deleted the move constructor and move assignment operator to prevent moving. These deletions enforce that there's truly only one instance at compile time.
The getInstance method implementation is elegant and thread-safe. It uses a static local variable that's created once and lives for the program's duration. This is guaranteed to be initialized only once, even in multithreaded environments, thanks to C++11's thread-safe static initialization. The lazy initialization means the instance is created only on first use, not at program startup.
In the main function, I can request the instance multiple times, and all references point to the same Game object. This provides a global access point to the game state from anywhere in the codebase without passing game pointers through every function call.
I chose the Singleton pattern for Game because only one game should run at a time in this application. It provides a convenient global access point for game state, manages the window, resources, and game loop in one place, and prevents accidental creation of multiple game instances that would conflict with each other.
The pattern ensures a single instance throughout program lifetime, provides global access through getInstance, uses a private constructor to prevent external instantiation, employs deleted copy and move operations to prevent duplication, and achieves thread-safe initialization with C++11 static locals. This makes it both convenient and safe to use throughout the codebase.
